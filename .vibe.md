## What is a Codemod?

A **codemod** (code modification) is an automated tool that performs large-scale code transformations. This framework allows developers to define code changes declaratively using YAML recipes, which are then executed against a codebase.

## Key Concepts

### Recipes
A **recipe** is a YAML file that defines a code transformation. Recipes consist of:
- Metadata (id, name, description, dslVersion)
- Arguments (args)
- Steps (edit, create, recipe composition)
- Maps (reusable data)
- Post-execution hooks

### AST Paths
**AST Paths** (Abstract Syntax Tree Paths) are used to navigate and identify code elements. They allow precise targeting of code locations for transformations.

Example AST path: `[class: MyClass, method: myMethod, arg: first]`

### Anchors
**Anchors** define insertion points relative to code elements. They determine WHERE new code should be inserted.

Available anchors:
- `doc:before` / `doc:after` - Relative to documentation comments
- `member:first` / `member:last` - Relative to class members
- `arg:first` / `arg:last` / `arg:index` - Relative to function arguments
- `param:first` / `param:last` - Relative to parameters
- `call:FunctionName` - Relative to function calls
- And many more (see DSL skill for complete list)

### Steps
Recipe steps define WHAT to do. Each step can be:
- **edit**: Modify existing files
- **create**: Create new files from templates
- **recipe**: Reference/compose other recipes

### Edit Operations
Within an edit step, you can:
- **insert**: Add new code at a specific location
- Other operations (see DSL skill)

## YAML DSL at a Glance

See the **yaml-dsl skill** for complete documentation, but here's a quick overview:

### Basic Recipe Structure
```yaml
dslVersion: 1
id: my_recipe
name: My Recipe
 description: What this recipe does

args:
  - name: file
    required: true
    inputKind: file
    help: The file to modify

steps:
  - edit:
      path: "{{file}}"
      steps:
        - insert:
            at:
              - class: MyClass
            anchor: member:last
            text: |
              void newMethod() {}

postExecution:
  - run: dart format .
```

### Key Fields

#### Recipe Level
- `dslVersion`: DSL version (currently 1)
- `id`: Unique identifier (required if no name)
- `name`: Human-readable name (required if no id)
- `description`: What the recipe does
- `args`: Recipe arguments
- `steps`: List of transformation steps
- `maps`: Reusable key-value mappings
- `postExecution`: Commands to run after execution

#### Arguments (args)
- `name`: Argument name (required)
- `required`: Whether argument is mandatory
- `help`: Help text
- `abbr`: Short abbreviation
- `contextKey`: Context key for auto-fill
- `inputKind`: Type of input (text, file, directory, symbol)
- `options`: Available options for dropdown selection
- `allowCustomValue`: Allow values not in options list
- `defaultsTo`: Default value

#### Steps
Each step can be one of:
- `recipe`: Reference another recipe by ID
- `edit`: Edit existing files
- `create`: Create new files

#### Edit Step Structure
```yaml
- edit:
    path: "{{file}}"  # File path (can use argument templates)
    steps:
      - insert:
          at: [class: MyClass, method: myMethod]  # AST path
          anchor: member:last  # Insertion point
          text: "void newMethod() {}"  # Code to insert
```

#### Insert Operation
- `at`: AST path (list or map format)
- `anchor`: Insertion anchor point
- `text`: Text/code to insert

#### Create Operation
```yaml
- create:
    path: "lib/new_file.dart"
    template: "file content here"
    # OR
    templateFile: "templates/new_file.dart"
    ifExists: skip  # or overwrite, error
    format: true  # Auto-format the file
```

#### Post-Execution
```yaml
postExecution:
  - run: dart format .
  - run: dart analyze
  - runScript: scripts/post_process.sh
```

## Coding Conventions

### Dart/Flutter Conventions
- Use `lowerCamelCase` for variables and functions
- Use `UpperCamelCase` for classes and types
- Use `lowercase_with_underscores` for file names
- Use `//` for single-line comments, `/* */` for multi-line
- Use `///` for documentation comments
- Prefer `final` for variables that don't change
- Use `const` for compile-time constants
- Follow Dart's effective Dart guidelines

### Project-Specific Conventions
- AST path components use `lowerCamelCase` (e.g., `class: myClass`)
- Anchor names use `lowercase:colon` format (e.g., `member:last`)
- Template variables use `{{variableName}}` syntax
- Recipe IDs use `lowercase_with_underscores`
- Test files use `_test.dart` suffix

## Common Patterns

### Pattern 1: Adding a Method to a Class
```yaml
steps:
  - edit:
      path: "{{file}}"
      steps:
        - insert:
            at:
              - class: "{{className}}"
            anchor: member:last
            text: |
              void {{methodName}}() {
                // implementation
              }
```

### Pattern 2: Adding a Field to a Class
```yaml
steps:
  - edit:
      path: "{{file}}"
      steps:
        - insert:
            at:
              - class: "{{className}}"
            anchor: member:last
            text: "final {{type}} {{fieldName}};"
```

### Pattern 3: Inserting at Function Call
```yaml
steps:
  - edit:
      path: "{{file}}"
      steps:
        - insert:
            at:
              - call: WidgetName
            anchor: arg:last
            text: ", key: ValueKey('{{key}}')"
```

### Pattern 4: Creating a New File
```yaml
steps:
  - create:
      path: "lib/{{name}}.dart"
      template: |
        class {{Name}} {
          // class content
        }
      format: true
```

## Error Handling

Common error types:
- **Parse errors**: Invalid YAML syntax
- **Validation errors**: Missing required fields, invalid structure
- **AST errors**: Cannot find specified code elements
- **Anchor errors**: Invalid anchor positions
- **Template errors**: Missing template variables
