# Transforms & Patches

## When to use

Use this skill when you are:
- Writing a new recipe transform that produces `SourcePatch` edits.
- Debugging preview output that looks wrong (offsets, wrong replacement, missing description).
- Fixing host/extension preview/diff mismatch caused by patch ordering or overlap validation failures.

## Where to look (core patch model)

- `SourcePatch`, patch application, overlap validation, and preview/diff helpers:
  - `lib/src/patch_helpers.dart`
    - `class SourcePatch`
    - `applyPatches(...)` (patch application order)
    - `validateNonOverlappingPatches(...)`
    - `previewPatches(...)`, `generateDiff(...)`
    - helpers like `insertAfterNode(...)`, `replaceNode(...)`, `deleteNode(...)`

- Deterministic patch validation:
  - `validateNonOverlappingPatches(...)` throws on overlaps except equal-offset insertions.

## Where to look (transform + operation plumbing)

- Transform interface:
  - `lib/src/transform.dart`
    - `abstract interface class CodeTransform`
    - `FunctionTransform` wrapper for function-based transforms

- Patch-producing operation:
  - `lib/src/operation.dart`
    - `EditDartFileOperation.collect(...)`
      - reads file source
      - runs each transform: `transform.apply(source, context)`
      - validates: `validateNonOverlappingPatches(patches)`
      - returns a `PatchFileChange` with the patches list order

## Patch shape (what the extension consumes)

In the host-to-extension JSON, a patch entry includes:
- `index`: position in the patch list generated by transforms
- `offset`: byte offset in the original source string
- `length`: number of bytes replaced/deleted
- `replacement` / `replacementPreview` depending on preview vs diff
- `description`: optional human label used by UI

On the TS side, this is represented by `PatchInfo` / `FilePreview`:
- `vscode_extension/src/types.ts` (`PatchInfo`, `FilePreview`)
- `vscode_extension/webview-ui/src/types.ts` (mirrors the same view models)

Preview vs diff mapping:
- Preview (snippet) uses:
  - `lib/src/vscode/diff_service.dart` with `includePatchReplacements: false`
  - emits `replacementPreview` (not full `replacement`) when configured
- Diff materialization uses:
  - `includePatchReplacements: true`
  - includes full replacement needed to render original vs modified content

## Critical invariants (don‚Äôt violate)

### 1. Offsets/lengths are relative to the original source

`applyPatches(source, patches)` assumes every patch references offsets in the *original* string.
When you compute offsets using analyzer AST nodes, make sure:
- patch offsets use the same unit (byte offsets)
- replacements are based on that same source snapshot passed into transforms

### 2. Patches must be non-overlapping

`validateNonOverlappingPatches` enforces:
- equal-offset insertions are allowed (they don‚Äôt overlap because length is 0)
- any overlapping non-empty edit ranges throw `StateError`

This is why transform ordering or patch ranges must be precise.

### 3. Deterministic patch ordering

Patch application uses a stable ordering:
- `applyPatches` sorts patches with a ‚Äústable‚Äù tie-break:
  - equal offset insertions apply in reverse input order so transforms‚Äô declaration order is preserved

If you generate patches with nondeterministic ordering, UI snippets and ‚Äúwhat you see vs apply‚Äù can drift.

## Patch application workflow (what happens at runtime)

Given a file source and a list of `SourcePatch`:

1. Validate overlap (`validateNonOverlappingPatches`)
2. Apply patches end-to-start (`applyPatches(...)`)
3. `PatchFileChange.generate()` produces modified source
4. `diff_service.dart` serializes:
   - preview snippet + replacement preview
   - or full original/modified content for diff

Where:
- `lib/src/operation.dart`: `PatchFileChange.generate()`
- `lib/src/patch_helpers.dart`: `applyPatches(...)`
- `lib/src/vscode/diff_service.dart`: serialization

## Implementation checklist

Before shipping a transform that emits patches:
- [ ] Confirm offsets/lengths are computed from the same `source` string passed to `transform.apply(source, context)`
- [ ] Ensure non-overlapping ranges (or only equal-offset insertions with `length = 0`)
- [ ] Set `description` to something stable and user-relevant (UI snippet headers)
- [ ] Keep transform output deterministic across runs
- [ ] If you create multiple patches at equal offset:
  - preserve intended order in the patch list so tie-break logic does the right thing

## Red flags

- Emitting overlapping patches (host will throw during collection).
- Emitting patches whose offsets refer to a mutated/intermediate source.
- Returning patches without stable ordering, causing:
  - snippet previews to appear inconsistent
  - diff preview mismatch
- Changing the meaning of `index` (it must remain ‚Äúindex in patch list‚Äù used by UI selection).

## Notes on adding new patch helpers

If you add helper methods in `patch_helpers.dart`:
- keep them pure and deterministic
- document whether they use offsets/lengths in bytes
- add unit tests for:
  - equal-offset insertions ordering
  - overlap validation behavior
  - preview formatting assumptions (if description-dependent)
